# デザイン概要

このページは socsim の概念的な入口です：socsim とは何か，その設計思想，コア型の役割，そして6フェーズループの1ステップで各パーツがどのように振る舞うかを詳細に説明します．リファレンスレベルの内容（クレート依存グラフ，キャリブレーション哲学，シナリオスキーマ）については[アーキテクチャ](architecture.ja.md)ページを，同梱のメカニズムについては[メカニズムカタログ](mechanisms.ja.md)を参照してください．

## 1. socsim とは

`socsim` はコンポーザブルなエージェントベース社会シミュレーションエンジンです．シミュレーションは，**メカニズム**のスタックが固定された6フェーズ順序でステップごとに変換する**共有ワールド**であり，すべての乱数は単一のシードRNGから生成されます．

基本的な比喩はニューラルネットワークです：メカニズムはレイヤーのように合成され，ワールドはそれらが読み書きするテンソルであり，エンジンはそれらを順番に実行するランタイムです．モデルの組み立ては，エンジンを編集するのではなく*メカニズムをスタック*することで行います．

![socsim at a glance](assets/design-overview.svg)

結果として，**ABM + RL スタイルのティックループ**が得られます：離散時間が1ステップずつ進み，すべてのエージェントとグローバル効果はフェーズにスロットされたメカニズムとして表現され，実行全体は指定したシードに対して決定論的です．

## 2. 設計思想

**修正ではなく合成．** ドメインロジックはメカニズムに宿ります．新しい挙動を追加するとは，もう1つの `Mechanism` を書いてフェーズにスロットすることを意味します — エンジン，ワールドトレイト，その他すべてのメカニズムはそのまま維持されます．これが「ニューラルネットのレイヤーのように合成する」の実践的な意味です．

**状態とコードは分離する．** 実行の*状態*はワールド（クロックを所有する），RNGストリームの位置，停止フラグです．*コード*はメカニズムのセット，スケジューラー，レコーダーです．[`Snapshot`](#7-状態とコードスナップショットと決定論性)は状態のみを捕捉します — PyTorch の `state_dict` と model architecture の分離と同様に — 実行を保存してビット単位で再開できます．

**構造による決定論性．** 再現性はオンにする機能ではなく，構造的なものです．すべての乱数は単一のシード ChaCha20 ストリームから派生し，エージェントと集計は常にソートされた `AgentId` 順で反復されて浮動小数点の和とRNGの引きが安定し，クロックは値渡しされるのでtの読み取りがワールドへの2回目のborrowを必要とせず，活性化順序はハッシュマップの順序に依存した偶発的なものではなく明示的な `Scheduler` の決定です．

**小さなトレイト面．** 実装すべきトレイトは `WorldState`，`Mechanism`，`Scheduler`，`Recorder` の4つのみで，ほとんどのプロジェクトは最初の2つしか書きません．それ以外はすべて具体的なエンジンの機構です．

**2つのファーストクラス利用経路．** 同じエンジンと決定論性の保証が，シナリオTOML / CLI経路（`.toml` にメカニズムを宣言し `socsim` バイナリで実行）とライブラリモード（Rustでワールドを構築して `Simulation` を直接駆動）の両方を支えます．どちらもセカンドクラスではありません．

**6フェーズループは協調のコントラクト．** メカニズムは互いを呼び出しません．代わりに，フェーズを宣言することで*いつ*効果が発生するかを合意します．その固定順序 — `PreStep → Environment → Decision → Interaction → Reward → PostStep` — が予測可能に合成するために必要な唯一のプロトコルです．

## 3. 登場人物

socsim はあなたが実装する4つのトレイトと，エンジンが提供するいくつかの具体的な型から構成されています．

![Core abstractions and ownership](assets/design-abstractions.svg)

| 型 | 種別 | 役割 | 比喩 |
|---|---|---|---|
| `WorldState` | トレイト（実装が必要） | 共有可変環境：エージェントロスター，`SimClock`，すべてのドメインデータ． | テンソル / モデル状態 |
| `Mechanism<W>` | トレイト（実装が必要） | 研究ロジックの1単位．実行するフェーズを宣言し，`apply` で処理を行う． | ネットワークレイヤー |
| `Scheduler<W>` | トレイト（実装が必要，または組み込みを利用） | 各ステップのエージェント活性化順序を決定する． | データローダーの順序 |
| `Recorder` | トレイト（実装が必要，または組み込みを利用） | メトリクスとイベントのシンク．`NullRecorder` がデフォルトのno-op実装． | ロガー / メトリクスライター |
| `AgentId` | struct | 不透明な `Copy` エージェントID；反復が決定論的になるよう `Ord` を実装． | 行インデックス |
| `SimClock` | struct | `Copy` の離散時間カウンター：`t`，`t_max`，`is_done`，`tick`． | エポックカウンター |
| `Phase` | enum | 6フェーズ；`Phase::ORDER` がその固定実行順序． | フォワードパスのスケジュール |
| `StepContext<'a, W>` | struct | すべての `apply` 呼び出しに渡されるボローのバンドル． | レイヤーのフォワード `ctx` |
| `Blackboard` | struct | ステップスコープの型消去スクラッチ；メカニズム間でトランジェントな値を渡す． | ステップごとの活性化キャッシュ |
| `SimRng` | struct | シード ChaCha20 ストリーム；`derive` でラベル付き子ストリームを作成する． | シードジェネレーター |
| `Simulation<W>` | struct | ドライバー．ワールド，メカニズム，スケジューラー，RNG，レコーダー，スクラッチ，停止フラグを所有する． | 学習ループ / ランタイム |
| `SimulationBuilder` | struct | 適切なデフォルト（シーケンシャルスケジューラー，シード0，`NullRecorder`）を持つフルーエントコンストラクター． | モデルビルダー |
| `Snapshot<W>` | struct | 保存/再開のための可変状態のシリアライズ可能なキャプチャ． | `state_dict` |

所有権はシンプルです：`Simulation` は各コラボレーターを1つずつ**所有**し，すべてのメカニズム呼び出しで `StepContext` を通じて一時的かつ安全に貸し出します．メカニズムはborrowするだけで，エンジン状態を所有することはありません．

## 4. 6フェーズ実行モデル

すべてのステップは同じ6フェーズを同じ順序で実行します．フェーズは*意味的な*スケジュールです — 効果が何であるかではなく，どの種類の効果が*いつ*属するかを示します：

- **PreStep** — メインフェーズ前のセットアップ / ブックキーピング．
- **Environment** — グローバルな外因性の更新（リソースの補充，ショック）．
- **Decision** — エージェントが行動を選択する（このモジュールでは採用/退職も含む）．
- **Interaction** — エージェント間の効果（ピア効果，ネットワーク拡散）．
- **Reward** — ペイオフの計算と集計；メトリクスの記録．
- **PostStep** — 全員が行動した後のクリーンアップ / ロギング．

`Simulation::step()` が行う処理を正確に示します：

![Anatomy of one step](assets/design-step-sequence.svg)

1. **クロックをティック**（`t += 1`）します — メカニズムが新しい時刻を観測できるよう*最初に*行います．
2. **クロックを値スナップショットにコピー**します — `StepContext` が `&mut world` を渡すため，クロックは値渡しされ，`t` の読み取りにワールドへの2回目のborrowが不要です．
3. **スクラッチのblackboardをクリア**します — 前のステップの値が今のステップに漏れないようにします．
4. **スケジューラーに活性化順序を問い合わせ**ます — 同じ `agent_order` がこのステップのすべてのフェーズで共有され，すべてのメカニズムが一貫した順序を見ます．
5. **ネストされたループを実行**します．外側のループは `Phase::ORDER` を走り，内側のループはメカニズムを**挿入順**（シナリオ宣言順と等しい）で走ります．メカニズムは `phases()` で登録したフェーズでのみ実行され，複数のフェーズを登録したメカニズムはそれぞれで1回ずつ呼び出されます．各呼び出しでエンジンは新鮮な `StepContext` を構築して `apply(phase, ctx)` を呼びます．
6. **返る．** 実行ループがもう1ステップ進むかどうかを決定します．

リファレンスHRライフサイクルメカニズムにマッピングすると，1ステップは次のように流れます：

| フェーズ | 動作するメカニズム | 何が起きるか |
|---|---|---|
| PreStep | — | （ブックキーピング用；MARLバリアントで使用） |
| Environment | `learning_curve` | 在職期間が加算され，個人生産性が更新される |
| Decision | `fit`, `turnover`, `hiring` | 満足度の更新；退職の解決；空席の補充 |
| Interaction | `peer_effect`, `ocb`, `toxic_spread` | チーム効果，知識流入，伝染 |
| Reward | `org_performance` | 生産性の合計；ステップメトリクスの記録；チーム平均の再計算 |
| PostStep | `knowledge_loss`, `socialization` | 退職者の暗黙知の消失；新入社員のオンボーディング |

順序が固定されているため，メカニズムは互いを参照することなくハンドオフに依存できます：`turnover` は Decision で `departed_this_step` に書き込み，`knowledge_loss` は PostStep でそれを消費します；`org_performance` は Reward で各チームの平均能力を再計算し，`peer_effect` は次のステップで最新の値を読み取ります．各メカニズムのコントラクトは[メカニズムカタログ](mechanisms.ja.md)を参照してください．

## 5. メカニズム呼び出しの内側：`StepContext`

メカニズムの `apply` が小さくフォーカスされているのは，触れられるすべてのものが1つのバンドルで届くからです：

![Anatomy of StepContext](assets/design-stepcontext.svg)

- `world: &mut W` — 共有状態を読み取り，変更する．
- `clock: SimClock` — コピー；現在の `t` を自由に読み取れる．
- `rng: &mut SimRng` — 乱数の*唯一の*正規ソース；実行が再現可能なままになる．
- `recorder: &mut dyn Recorder` — メトリクスとイベントを出力する．
- `agent_order: &[AgentId]` — スケジューラーからのこのステップの活性化順序．
- `scratch: &mut Blackboard` — ステップスコープの型消去スペース；同じステップ内の後のメカニズムに（または外部のドライバーに）トランジェントな値を渡すためのもので，`WorldState` を汚染しない．
- `stop: &mut bool` — `request_stop()` を呼んで現在のステップの完了後に実行を終了する．

これがメカニズムをコンポーザブルに保つ仕組みです：メカニズムは「ワールドとこのコンテキスト」の純粋な関数であり，隠れたグローバル状態も互いへの直接的な知識もありません．

## 6. シミュレーションの駆動

`Simulation` は `step()` に加えて小さな駆動APIを提供します：

- `run()` — クロックが `is_done()`（`t ≥ t_max`）になる**か**メカニズムが停止を要求するまでステップを進める．
- `run_until(predicate)` — `predicate(&world)` が真になったときにも停止する．
- `run_observed(observe)` — 各ステップの後に `observe(StepReport)` を呼んで収束曲線やライブメトリクスを収集する；`step()` + 読み取りループを自分で組む必要がない．
- `step()` / `step_reported()` — ループを自分で駆動したいときに正確に1ステップ進める．

ステップの途中で要求された停止は，ステップの完了後に*尊重*されます：現在のステップの残りのメカニズムは引き続き実行され，その後ループが終了します．`StepReport` はステップ後のクロック時刻，停止フラグ，ワールドとスクラッチへの共有参照をまとめます．

```rust
use socsim_config::{Registry, Params, ModulePack};
use socsim_hr_lifecycle::{HrLifecyclePack, HrWorld};
use socsim_engine::{RandomActivationScheduler, SimulationBuilder};

let mut reg: Registry<HrWorld> = Registry::new();
HrLifecyclePack.register(&mut reg);

let mut builder = SimulationBuilder::new(world)
    .scheduler(Box::new(RandomActivationScheduler))
    .seed(42);
for name in ["learning_curve", "fit", "turnover", "hiring",
             "peer_effect", "ocb", "toxic_spread",
             "org_performance", "knowledge_loss", "socialization"] {
    builder = builder.add_mechanism(reg.build(name, &Params::empty())?);
}
let mut sim = builder.build();
sim.run()?;
```

カスタムワールドとメカニズムの構築については[ライブラリAPI](library.ja.md)を，同じエンジンをシナリオファイルから駆動する方法については[CLIリファレンス](cli.ja.md)を参照してください．

## 7. 状態とコード：スナップショットと決定論性

`Snapshot<W>` は socsim の保存/再開プリミティブであり，状態とコードの分離を体現しています：

![State vs code](assets/design-state-vs-code.svg)

**状態のみ**を捕捉します — ワールド（`SimClock` を所有する），正確な `SimRng` ストリーム位置，停止フラグ．**コード** — メカニズム，スケジューラー，レコーダー — は意図的に省略します；これらは `Simulation` を再構築するときに再度供給されます．*同じ*メカニズムで構築した simulation にスナップショットを復元すると，保存したステップからビット単位で実行が継続します；`Snapshot::save` / `load` はバージョンチェック付きJSONとして永続化します．

これが機能するのは，エンジン全体が決定論的だからです：単一のシード ChaCha20 ストリーム（ラベル付き `derive` 子を持つ），安定した浮動小数点順序のためのソートされた `AgentId` 反復，値コピーされたクロック，明示的なスケジューラー．同じシード，同じコード，同じ軌跡 — どのマシンでも．
